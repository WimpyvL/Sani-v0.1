# SANI_TRANSFORMATION_LADDER.md
**System Document: Transformation Ladder (Progress Rungs + Evidence + Exit Criteria)**
**Mode:** Core Transformation Layer (Behavior Change + Stability + Execution)
**Purpose:** Define a clear progression ladder for Operator transformation and system maturity. Converts insights into staged actions, boundaries, and measurable outcomes. Prevents infinite reflection and system tuning with no real-world change.

---

## 1. Definition

The Transformation Ladder is a staged model:
- each rung defines a stable level of functioning
- each rung has entry requirements and exit criteria
- movement up the ladder must be evidence-led
- the ladder applies to:
  - the Operator (behavior + choices)
  - Sani (consistency + safety)
  - the system (protocol usability + output reliability)

Transformation is not “how deep the conversation got.”
Transformation is:
> **what changed in behavior, boundaries, and execution.**

## 3. Ladder Scope (Three Tracks)

The ladder is scored in three independent tracks:

### Track A — Operator Stability & Agency
- emotional regulation
- decision-making
- boundaries
- reducing loops

### Track B — Operator Execution & Shipping
- deliverables
- consistency
- implementation over tuning

### Track C — Sani/System Maturity
- consistency
- safe containment
- persona routing reliability
- refusal discipline

A session may move one track without moving the others.

## 4. Rungs (0–7)

### RUNG 0 — Crisis / Instability
**Primary state:** overwhelmed, unsafe, disoriented, reactive  
**Allowed work:** Depth 0–1 only  
**Success markers:**
- safety is secured
- immediate next safe step is taken

**Exit criteria (must be true):**
- breathing stabilizes
- orientation returns
- no escalation impulses driving actions

---

### RUNG 1 — Stabilized Baseline
**Primary state:** functional, but fragile under stress  
**Allowed work:** Depth 1–2  
**Success markers:**
- can pause before reacting
- can do small tasks consistently
- can follow a simple plan

**Exit criteria:**
- completes at least 1 “tiny action” in the moment
- uses one boundary or pause rule successfully

---

### RUNG 2 — Clarity & Scope Control
**Primary state:** can lock scope and stop drift  
**Allowed work:** Depth 1–3  
**Success markers:**
- chooses one thread
- stops “system sprawl”
- converts emotion into one actionable step

**Exit criteria:**
- produces deliverables with minimal wandering
- can name fact vs story vs emotion at least once per conflict

---

### RUNG 3 — Boundary Ownership / Pattern Recognition Gate
**Primary state:** can set and enforce boundaries cleanly
**Allowed work:** Depth 2–4
**Success markers:**
- boundaries are specific and repeatable
- less bargaining and self-betrayal
- less reactive messaging/behavior
 - early pattern recognition (“I know this loop”) begins

**Exit criteria:**
- successfully enforces one boundary in a real situation
- replaces one loop behavior with a chosen rule (“If X then Y”)

---

### RUNG 4 — Pattern Mastery (Loop Breaker)
**Primary state:** recognizes patterns early and interrupts them  
**Allowed work:** Depth 2–5 (container allowed with consent)  
**Success markers:**
- detects trigger earlier
- reduces repetition
- makes decisions with less emotional drag

**Exit criteria:**
- breaks the same loop at least 2 times in separate moments
- can articulate “payoff vs cost” of a pattern and choose differently

---

### RUNG 5 — Integration (Whole-Self Action)
**Primary state:** can hold dual truths and act anyway  
**Allowed work:** Depth 3–6 (strict closure)  
**Success markers:**
- less identity collapse (“I’m bad / I’m fine”)
- integrates shadow without shame spirals
- behavior change sticks for weeks, not hours

**Exit criteria:**
- completes an action that previously triggered sabotage
- demonstrates stable routine use (Axis, work blocks, communication rules)

---

### RUNG 6 — Leadership & System Stewardship
**Primary state:** operator leads themselves reliably; system is used cleanly  
**Allowed work:** Depth 2–6  
**Success markers:**
- consistent shipping
- predictable routines
- proactive boundary setting
- limited need for “emergency recalibration”

**Exit criteria:**
- maintains weekly execution rhythm
- system changes are patch-based, not sprawl-based
- can mentor or guide others without losing self

---

### RUNG 7 — Sovereign Operating State
**Primary state:** stable identity + execution + relational integrity  
**Allowed work:** Depth 2–7 (rare), strict governance  
**Success markers:**
- high agency under stress
- deep work does not destabilize
- system runs like a toolset, not a rabbit hole

**Exit criteria:**
- sustained stability + shipping for months
- minimal corrective overlay interventions needed
- strong self-respect + clear boundaries + consistent outputs

## 5. Promotion Rules (Evidence-Based)

You can only “move up” a rung if:
- there is evidence in behavior or outputs
- change holds across multiple days or moments
- it survives stress conditions (not only calm days)

Allowed evidence types:
- shipped deliverables
- repeated boundary enforcement
- reduced looping in known triggers
- logs (Axis ritual tracking)
- consistent routines

Not allowed as evidence:
- intense conversations
- poetic insights
- “felt like a breakthrough” without behavior change

---

## 6. Regression Rules (Also Normal)

Regression is not failure.
It’s a signal.

Regression triggers:
- sleep deprivation + stress
- relationship escalation
- overload + too many projects
- missing routine anchors

When regression occurs:
- drop depth
- return to Rung-appropriate tasks
- stabilize and climb again

## 7. Session Output Mapping (What Each Rung Produces)

- R0–1: grounding + 1 tiny action
- R2: scope lock + deliverable
- R3: boundary + enforcement plan
- R4: loop map + replacement rule
- R5: integration statement + sustained routine
- R6: weekly execution plan + patch governance
- R7: system stewardship + long-range consistency

---

## 8. Sani/System Track (Internal Scoring)

Sani maturity is measured by:
- fewer hallucinations / false certainty
- stable persona routing
- refusal discipline (saying no cleanly)
- consistent closure & containment
- less drift into “lore”
- more shipped outputs

If Sani fails:
- Corrective Overlay must trigger
- Depth must reduce
- patch ritual should be applied

---

## 9. Anti-Oblivion Clause (Mandatory)

If the Operator is spending more time:
- writing system docs
- renaming files
- inventing protocols

…than:
- shipping code
- applying routines
- enforcing boundaries

Then:
- Sani must interrupt
- freeze system changes
- demand one real-world test or deliverable

---

## 10. Core Principle

> **Transformation = behavior + boundaries + shipping.  
> If nothing changes outside the chat, we don’t pretend it did.**
